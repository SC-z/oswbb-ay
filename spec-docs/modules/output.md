# 模块 Spec

## 1. 模块概览
- 模块名称：output
- 所属子系统：输出层
- 架构位置：数据转换与报告导出
- 一句话职责描述：将解析结果转换为统一指标并输出为 CSV/JSON/HTML。

## 2. 设计背景与问题域
- 背景说明：用户需要不同格式的结果用于二次分析或分享。
- 需要解决的问题：统一导出接口、维持数据字段一致性。
- 约束条件：report 模式不走此模块。

## 3. 设计目标与非目标
- 设计目标：
  - 提供 `OutputFormatter` 接口统一输出方式。
  - 保证导出字段稳定可用。
- 非目标：
  - 不负责分析逻辑或异常判定。

## 4. 核心职责与边界
- 核心职责：
  - 将 `*Log` 转换为 `*RawMetrics`。
  - 输出 CSV/JSON/HTML 文件。
- 职责边界说明：不解析源文件，不做统计分析。
- 不应承担的责任：流程编排或日志分类。

## 5. 架构位置与协作关系
- 上游模块：`processor`、`iostat/meminfo/top`。
- 下游模块：无（输出到文件系统）。
- 依赖方向说明：只依赖解析结果与格式化器实现。

## 6. 核心设计与关键流程
- 抽象设计说明：`CreateFormatter` 根据格式选择实现。
- 关键流程描述：
  - `Convert*Data` 生成行式指标数组。
  - `CSV/JSON/HTML` formatter 写文件。

## 7. 关键实现说明
- 核心接口 / 数据结构：`OutputFormatter`、`*RawMetrics`。
- 实现要点：
  - `ml` 格式等价于 `csv`。
  - HTML 以模板内嵌 JSON，依赖 ECharts CDN。
- 与设计意图的映射：统一输出接口，易扩展新格式。

## 8. 设计取舍与已知限制
- 设计取舍：report 继续在 `processor` 输出，避免模板化改造。
- 已知限制与风险：
  - HTML 离线环境可能无法加载图表。
  - 字段变化需同步多格式输出。

## 9. 演进方向与重构提示
- 可能的演进方向：
  - 增加本地静态资源版 HTML。
  - 增加 Parquet/Excel 等导出。
- 修改注意事项：保持 `*RawMetrics` 字段一致性。
