# 模块 Spec

## 1. 模块概览
- 模块名称：iostat
- 所属子系统：日志解析层
- 架构位置：解析与指标提供
- 一句话职责描述：解析 iostat 日志并提供 I/O 指标与趋势查询。

## 2. 设计背景与问题域
- 背景说明：iostat 输出版本多样，需要兼容不同列数。
- 需要解决的问题：从原始文本中抽取时间戳、CPU 与设备指标。
- 约束条件：以表头驱动解析，兼容 15/20 列格式。

## 3. 设计目标与非目标
- 设计目标：
  - 兼容多版本 iostat 列名差异。
  - 输出统一结构体用于上层分析与导出。
- 非目标：
  - 不负责报告输出与异常检测阈值制定。

## 4. 核心职责与边界
- 核心职责：解析日志、提供设备指标与趋势查询。
- 职责边界说明：只做解析与数据结构维护，不处理输出格式。
- 不应承担的责任：文件扫描、输出生成、跨模块调度。

## 5. 架构位置与协作关系
- 上游模块：`processor`。
- 下游模块：`common`（时间序列工具）。
- 依赖方向说明：解析结果被 `processor` 与 `output` 使用。

## 6. 核心设计与关键流程
- 抽象设计说明：先解析表头建立列索引，再按列名读取字段。
- 关键流程描述：
  - 识别时间戳行 `zzz ***` → 创建快照。
  - 解析 CPU 行 → 填充 `CPUStats`。
  - 解析 Device 表头 → 建立索引。
  - 解析设备数据行 → 生成 `DeviceStats`。

## 7. 关键实现说明
- 核心接口 / 数据结构：`IOStatParser.ParseFile`、`IOStatLog`、`DeviceStats`。
- 实现要点：
  - 表头驱动优先，若缺失则回退到 15/20 列固定格式。
  - 跳过 `md*` RAID 设备以避免噪声。
- 与设计意图的映射：以兼容为主，减少对固定列序的依赖。

## 8. 设计取舍与已知限制
- 设计取舍：忽略 `svctm`/`%util` 等字段，聚焦 I/O 与延迟。
- 已知限制与风险：
  - 新格式变更可能导致字段解析失败。
  - 时间戳固定解析为 CST。

## 9. 演进方向与重构提示
- 可能的演进方向：
  - 引入更完整的字段映射与版本检测。
  - 提供可配置的设备过滤策略。
- 修改注意事项：保持表头驱动解析为第一优先级。
