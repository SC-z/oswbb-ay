# 模块 Spec

## 1. 模块概览
- 模块名称：cli
- 所属子系统：接口层
- 架构位置：系统入口
- 一句话职责描述：解析 CLI 参数并触发处理流程。

## 2. 设计背景与问题域
- 背景说明：用户通过命令行执行离线分析。
- 需要解决的问题：收集输入路径、时间范围、输出格式等参数并交给处理器。
- 约束条件：保持简单、无业务逻辑、失败时快速退出。

## 3. 设计目标与非目标
- 设计目标：
  - 参数清晰、默认值合理。
  - 输入缺失时给出明确错误。
- 非目标：
  - 不负责任何解析、统计或输出逻辑。

## 4. 核心职责与边界
- 核心职责：flag 解析、参数校验、初始化时区、调用 `processor`。
- 职责边界说明：不接触文件系统、日志解析与输出。
- 不应承担的责任：日志格式判断、异常检测、输出文件生成。

## 5. 架构位置与协作关系
- 上游模块：用户/终端。
- 下游模块：`pkg/processor`。
- 依赖方向说明：单向调用，不被其他模块依赖。

## 6. 核心设计与关键流程
- 抽象设计说明：以 CLI 参数作为唯一输入源。
- 关键流程描述：解析参数 → 校验 `-f` → 初始化 CST 时区 → 调用 `ProcessPath`。

## 7. 关键实现说明
- 核心接口 / 数据结构：`flag` 包的参数指针。
- 实现要点：错误即 `log.Fatal` 直接退出。
- 与设计意图的映射：保证入口层保持最小职责。

## 8. 设计取舍与已知限制
- 设计取舍：入口逻辑极简，缺少更细的参数校验提示。
- 已知限制与风险：时区固定为 CST，未提供可配置入口。

## 9. 演进方向与重构提示
- 可能的演进方向：增加 `--timezone` 或自动读取本地时区。
- 修改注意事项：保持入口无业务逻辑，避免侵入 `processor`。
