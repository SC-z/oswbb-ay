# 模块 Spec

## 1. 模块概览
- 模块名称：processor
- 所属子系统：核心编排层
- 架构位置：主流程控制中心
- 一句话职责描述：扫描输入、合并日志、控制分析流程与输出。

## 2. 设计背景与问题域
- 背景说明：支持多日志类型与多输出格式，需要统一的流程编排。
- 需要解决的问题：文件识别、合并策略、时间过滤、报告/导出切换。
- 约束条件：日志格式与输出模块解耦，流程可扩展。

## 3. 设计目标与非目标
- 设计目标：
  - 支持文件与目录输入。
  - 统一处理时间范围与输出模式。
  - 尽量减少解析模块之间的耦合。
- 非目标：
  - 不直接写 CSV/JSON/HTML（交由 `output`）。

## 4. 核心职责与边界
- 核心职责：
  - 路径扫描与分类（iostat/meminfo/top）。
  - 单文件/合并模式切换。
  - 调用解析器并生成 report 或导出文件。
- 职责边界说明：不解析具体字段格式；不维护输出模板。
- 不应承担的责任：解析格式细节、输出模板逻辑。

## 5. 架构位置与协作关系
- 上游模块：`cli`。
- 下游模块：`iostat`、`meminfo`、`top`、`output`、`common`。
- 依赖方向说明：`processor` 负责调度解析与输出，依赖集中。

## 6. 核心设计与关键流程
- 抽象设计说明：以 `executeAnalysisTemplate` 为统一流程骨架。
- 关键流程描述：
  - `ProcessPath` 判断文件/目录 → 扫描与分类 → 可选解压 `.gz` → 单文件或按主机合并 → 解析与分析 → report 打印或导出。

## 7. 关键实现说明
- 核心接口 / 数据结构：
  - `FileProcessor.ProcessPath/ProcessDirectory/ProcessSingleFile`
  - `analysisOptions` 与 `executeAnalysisTemplate`
- 实现要点：
  - 文件类型通过文件名包含关键字判断。
  - 合并模式依赖文件名的主机名分组规则。
  - 报告模式绕过 `output`，直接打印分析结果。
- 与设计意图的映射：统一流程、不同日志类型复用模板。

## 8. 设计取舍与已知限制
- 设计取舍：report 与导出分流，导致 report 逻辑集中在 `processor`。
- 已知限制与风险：
  - 文件类型识别依赖文件名关键字。
  - `gzip` 解压调用外部命令，环境缺失会失败。
  - 时间范围解析固定格式，错误时直接返回。

## 9. 演进方向与重构提示
- 可能的演进方向：
  - 引入统一解析接口与注册机制。
  - 将 report 输出拆分为独立模块以减少 `processor` 体积。
- 修改注意事项：保持 `executeAnalysisTemplate` 的通用性与输出分流逻辑。
