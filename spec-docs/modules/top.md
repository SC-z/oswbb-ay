# 模块 Spec

## 1. 模块概览
- 模块名称：top
- 所属子系统：日志解析层
- 架构位置：解析与 CPU/Load 指标提供
- 一句话职责描述：解析 top 日志并输出负载、CPU、进程状态指标。

## 2. 设计背景与问题域
- 背景说明：top 输出格式相对稳定，但时间戳存在轻微变体。
- 需要解决的问题：从文本中提取 Load、CPU、Tasks 指标。
- 约束条件：依赖行前缀匹配，需容忍格式差异。

## 3. 设计目标与非目标
- 设计目标：
  - 解析关键性能指标并形成快照序列。
  - 适配不同 OSWatcher 时间戳空格格式。
- 非目标：
  - 不负责复杂的进程级分析或输出。

## 4. 核心职责与边界
- 核心职责：解析日志并维护 `TopSnapshot` 序列。
- 职责边界说明：仅提供数据，不参与输出与告警策略。
- 不应承担的责任：报告生成、文件扫描、合并逻辑。

## 5. 架构位置与协作关系
- 上游模块：`processor`。
- 下游模块：`common`（时间范围接口）。
- 依赖方向说明：解析结果被 `processor` 与 `output` 使用。

## 6. 核心设计与关键流程
- 抽象设计说明：以时间戳行作为快照边界，逐段解析关键行。
- 关键流程描述：
  - `zzz ***` 行解析时间戳并创建快照。
  - `load average` 行解析 Load1/5/15。
  - `Tasks:` 行解析任务状态。
  - `Cpu(s):` 行解析 CPU 百分比。

## 7. 关键实现说明
- 核心接口 / 数据结构：`TopParser.ParseFile`、`TopSnapshot`。
- 实现要点：
  - 兼容 `zzz ***Wed` 与 `zzz *** Wed` 两种格式。
  - 通过 `Sscanf` 与字段遍历解析指标。
- 与设计意图的映射：保证解析鲁棒性而不引入复杂依赖。

## 8. 设计取舍与已知限制
- 设计取舍：仅保留 OSWatcher 常用指标，忽略进程列表。
- 已知限制与风险：
  - 时间戳解析依赖字段位置，极端格式可能失败。
  - 未验证多语言 top 输出。

## 9. 演进方向与重构提示
- 可能的演进方向：
  - 引入更稳健的时间戳解析策略。
  - 支持解析更多 CPU 字段或进程摘要。
- 修改注意事项：保证快照边界正确，避免数据混淆。
